# V3.3 Plan: MCTS-Only Narrowed Thesis

**Created**: 2026-05-04 15:18:07 KST
**Status**: Proposed (not yet started)
**Trigger**: Current framework (greedy + RL × 5 + MCTS × 3 + BookSim selector) is criticized as a "kitchen-sink ensemble." Reviewer-friendly framing requires a single dominant search algorithm. Empirical pilot (K16_N4 cell3) showed that current MCTS settings alone lose to RL on 8/11 cases (mean +21 lat gap, max +104 lat). To make MCTS-only viable we need a stronger search budget plus possibly an upgraded surrogate.

## Thesis (proposed)

Multi-workload joint chiplet NoI superset can be obtained by Monte Carlo Tree Search alone (no policy gradient agent), provided the search budget is sufficient and the surrogate is accurate on dense joint workloads. The framework consists of:

```text
   Stage 1a — MCTS-only candidate generation
   Stage 1b — BookSim-based selection (now selecting between MCTS seeds only)
   Stage 2  — per-workload BookSim-greedy mask (unchanged)
```

Baselines: Mesh, Kite-S/M/L, GIA at iso-wire W. PARL excluded.

## Required upgrades

### 1. Stronger MCTS configuration

| Parameter | Current | Proposed |
|---|---|---|
| n_seeds | 3 | **7** |
| n_iters per seed | 1500 | **5000** |
| rollout_depth | 12 | **20** |
| expansion_branch | 25 | **40** |
| rollout_branch | 8 | **12** |

Per-seed MCTS cost estimate: current 10 min → proposed ~55 min.
Per-combo total MCTS cost: 7 × 55 = ~6.4 h (vs current ~30 min).

### 2. Multiple warm-starts (no RL, no kite_l)

Current uses only greedy_union as warm-start. Proposed adds diversity without
introducing baseline leakage:

- 3 seeds from greedy_union (deterministic per-W greedy + union)
- 2 seeds from random algorithmic spine (random hop-3 spine, randomly paired)
- 2 seeds from random uniform link sample within wire budget

Total: 7 MCTS seeds with 3 warm-start types. Each warm-start is generated by
the framework, never copied from a baseline (Mesh, Kite, GIA).

### 3. Surrogate accuracy decision

Current rate-aware surrogate is OOD on dense joint workloads. Three options:

- **3a. Keep current surrogate, longer MCTS**: try first. If MCTS-only with 7
  seeds × 5000 iters matches RL+MCTS combo, current surrogate is sufficient.
- **3b. Re-train surrogate** on dense joint workload data. Requires ~10k
  BookSim runs (~3 days compute) + retraining (~1 day). Total: ~4-5 days.
- **3c. BookSim-in-the-loop MCTS for top-K leaves**: surrogate for tree
  rollouts, BookSim for terminal verification on top-50 leaves. Slow but
  accurate. ~3-5 hours per combo extra.

Default decision: try **3a first** (cheapest, fastest). If it fails on the
hard cells (K16_N8 dense 3-mix, K32_N8), fall back to 3b or 3c.

## Validation pilot before full sweep

Before redoing all 44 combos × 4 cells, run a focused pilot on K16_N4 (already
done with current framework) and compare:

1. Run MCTS-only stronger on all 11 K16_N4 subsets at W=240.
2. Compare to current RL+MCTS+greedy results (already in JSON).
3. Success criterion: MCTS-only ≤ current RL+MCTS in at least 9/11 cases (gap
   ≤ 5%) on raw latency.

If pilot succeeds, run full sweep (4 cells × 11 subsets × W = 44 + 24 wire
points). If pilot fails, fall back to 3b (re-train surrogate).

## Migration plan from current sweeps

1. Let current sweeps continue running (they will finish even if we plan a new
   variant). Their results provide the **comparison baseline** for the
   MCTS-only thesis.
2. Implement the new modules above in parallel. No execution until current
   sweeps free up compute.
3. When current sweeps finish (or after cell0 K32_N8 timeout), launch
   K16_N4 pilot first.
4. Decide based on pilot outcome.

## Risk register

- **Risk: MCTS-only with stronger budget still loses to RL on some cells.**
  Mitigation: surrogate retraining (option 3b). Cost: ~5 more days.
- **Risk: Pilot succeeds on K16_N4 but fails on K32_N8 (high traffic).**
  Mitigation: per-cell decision; allow K=32 cells to use a different MCTS
  config (more iterations).
- **Risk: Reviewer asks for ablation showing each MCTS upgrade separately.**
  Mitigation: K16_N4 pilot already provides ablation if multiple MCTS configs
  are tested side by side.

## Decision criteria for proceeding

- Pilot K16_N4 ≥ 9/11 match: proceed with full sweep.
- Pilot K16_N4 < 9/11 match: stop, switch to surrogate retraining (3b).

## Paper-side narrative shift

Current framing: "REINFORCE + MCTS + greedy + selector ensemble"
Proposed framing: **"MCTS-based joint multi-workload optimizer"**

Paper structure:
- §III. Joint multi-workload formulation (unchanged).
- §IV. **MCTS over surrogate** for superset search; design choices
  (warm-start diversity, exploration-exploitation, top-K BookSim verification).
- §V. Per-workload runtime mask (Stage 2, unchanged).
- §VI. Iso-wire experimental evaluation; baselines: Mesh, Kite-S/M/L, GIA.
  REINFORCE-only ablation included to demonstrate MCTS necessity.

REINFORCE becomes a baseline ablation, not part of the framework. Cleaner
algorithmic story.

## Open questions

- Should the MCTS UCB constant `c` be tuned? Currently `c=1.4` (sqrt(2)). At
  larger budgets, smaller `c` (more exploit) may be better. Pilot ablation
  worth running.
- Should warm-start include random generators, or only greedy_union? More
  diversity = more search space coverage but possibly more variance.
- For K=32 cells where surrogate is least accurate, should we use option 3c
  (BookSim-in-loop verification on top leaves) selectively?
